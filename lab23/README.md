# Звіт з Лабораторної роботи №23 (ISP & DIP)
**Варіант 4:** Document Multifunction Machine

## 1. Аналіз порушень до рефакторингу
* **Порушення ISP:** Початковий інтерфейс `IMultiFunctionDevice` містив методи `Print`, `Scan` та `Fax`. Якщо клієнту потрібен був лише друк, він все одно залежав від методів сканування та факсу, що змушувало реалізовувати непотрібну логіку (або кидати `NotImplementedException`).
* **Порушення DIP:** Клас `OldSmartMachine` (або клієнтські класи) створював екземпляри модулів через оператор `new`. Це створювало жорстку залежність від конкретних класів, а не абстракцій.

## 2. Опис змін
* **ISP:** Інтерфейс розділено на три окремі: `IPrinter`, `IScanner`, `IFax`. Тепер класи реалізують лише те, що їм дійсно потрібно.
* **DIP/DI:** Впроваджено Dependency Injection через конструктор у класі `DocumentWorker`. Клас тепер залежить від інтерфейсу `IPrinter`, а не від конкретного заліза.

## 3. Висновки
Застосування ISP дозволило зробити інтерфейси "вузькими" та зрозумілими. Застосування DIP через DI (через конструктор) дозволило легко підміняти реалізації принтерів без зміни логіки самого воркера, що значно полегшує автоматичне тестування.
