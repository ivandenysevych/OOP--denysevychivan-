# Звіт з Лабораторної роботи №22 (LSP)
**Варіант 4:** Робота з файлами (File & ReadOnlyFile)

## 1. Аналіз порушення LSP
У початковій ієрархії клас `ReadOnlyFile` успадковується від `File`. 
Базовий клас `File` має метод `Write()`, що створює **контракт**: "будь-який об'єкт типу File підтримує запис".
Клас `ReadOnlyFile` порушує цей контракт, викидаючи `InvalidOperationException` при спробі запису. 

**Проблема:** Клієнтський код, який приймає об'єкт `File`, не знає про це обмеження. Підстановка `ReadOnlyFile` замість `File` ламає поведінку програми (Runtime Crash).

## 2. Альтернативне рішення (Зміна ієрархії)
Для рефакторингу було обрано підхід **розділення інтерфейсів (Interface Segregation)**:
- Створено інтерфейс `IReadable` для операцій читання.
- Створено інтерфейс `IWritable` для операцій запису.
- Клас `SecureReadOnlyFile` реалізує лише `IReadable`.

## 3. Висновки
Дотримання LSP дозволяє уникнути помилок під час виконання програми. Використання інтерфейсів замість глибокого наслідування дозволяє клієнтському коду чітко знати, які операції підтримує об'єкт, ще на етапі компіляції.
